The Harness Effect

How Orchestration Design Sets the Token Economics
of Enterprise Agentic AI

arXiv:2607.06906 2026-07-08 cs.AI

Writer Inc. · 28 位作者联名  |  工业经验报告 / 企业 Agent 系统

开场：直接说"今天不讲模型，讲编排层"。这是来自 Writer 的 28 人工业经验报告，把"被忽视的最大杠杆"指认为 harness——不是模型。

TL;DR · 一句话核心

在企业 agentic AI 系统里，harness（编排层）比模型本身更能决定单任务的 token 账单。 把传统生产循环换成 Writer Agent Harness，跨 6 个模型平均：

−41%单任务成本 ($/task)
−44%端到端延迟 (s/task)
−38%单任务 tokens

而 质量 n=22 上统计持平（0.78 → 0.81，p 不显著）。
"模型升级是 N×1 的加法，harness 升级是 N×M 的乘数"——同一套编排改造，对每个未来模型都生效。

先把三个减法甩出来：成本 -41%、延迟 -44%、tokens -38%，质量没动。读者立刻能听出"这不是在牺牲质量换便宜"。

背景 · Token Maxing (Jevons 悖论)

行业默认做法是"花更多 token 换能力"，四条都在加码：

  • 长推理 · long CoT、单步多次采样、自我反思
  • 多轮对话 · k 步 agent loop、用户往返纠错
  • 宽 schema · 工具目录全量注入、retrieval 灌进 system
  • 全量历史重放 · 每次 turn 把整段 transcript 再喂一遍

单 token 价格 ↓ · 单任务 token 数 ↑ 更快

Anthropic 报告多 agent 系统 ~15× chat token；本文 100:1 input:output 比——Jevons 悖论在企业账单里照常发生。

Token maxing 是论文自造词，描述"边际 Q/τ 持续下降"的开发轨迹。先把行业共识放出来，下面才好说"没人动对的地方"。

问题 · 优化都打在"调用内部"

已有的 token 优化路径：

✓ 调用内部（in-call）

  • Prompt 压缩 / 摘要历史
  • Budget reasoning / CoT 限长
  • 路由到更便宜的模型
  • Speculative decoding、kv cache
  • 结构化输出约束 token 数

✗ 没人动：编排层 (orchestration)

  • Prompt 的两区结构
  • 历史压缩与 offload
  • 工具 schema 的 scope
  • 子代理与重试策略
  • Cache-aware 写入

论文的提问：如果把 token 经济学提到 orchestration 层，杠杆是否被低估了一个数量级？

用一个二分表格让"in-call 优化"和"orchestration 优化"对比一下。读者立刻看出论文的"空缺定位"。

核心公式 · Eq. 2 · 五项分解

把单次 turn 的输入 token \(T_{in}\) 完全展开：

$$T_{in} \;=\; S \;+\; H \;+\; G \;+\; R \;+\; U$$
SSystem
HHistory
GTool schemas
RRetrieval
UUser turn

单任务账单（k 轮 loop）：
\(\;C \;=\; \sum_{i=1}^{k}\big(p_{in} \cdot T_{in}^{(i)} \;+\; p_{out} \cdot T_{out}^{(i)}\big)\)

S/H/G/R/U 五项全部可被 harness 改写，U 是唯一"真用户信号"。这是把"in-call 优化"重定位为"harness 优化"的关键一步。

这是论文最重的一页。Eq. 2 的形式化把"token 经济"从直觉升级为公式。强调 S/H/G/R/U 都是 harness 可控的，U 才是用户信号。

核心公式 · Eq. 4 · 有效输入价

把 cache 折扣折进价格：

$$p_{in}^{eff} \;=\; p_{in} \cdot \big(1 \;-\; h \cdot (1 - \kappa)\big)$$

其中 \(h\) = cache 命中率，\(\kappa \approx 0.1\) = cached token 折扣比。

当 \(h \to 1\)，\(p_{in}^{eff} \to \kappa \cdot p_{in} \approx 0.1 \cdot p_{in}\)——harness 控制 h。

Figure 2 · Two-Zone Prompt

把 prompt 切成两个区： byte-stable prefix（工具 schema 目录、稳定 system、append-only 写入区） + 可变尾部（当轮新检索/新事件）

缓存命中靠"前缀字节稳定"——不把时间戳、UUID、新检索结果塞进前缀，h → 1，\(p_{in}^{eff} \to 0.1 \cdot p_{in}\)。
这是把"cache hit rate"从工程黑话升级为可设计参数的关键图。

Eq. 4 的"h 接近 1"是怎么做到的？靠两区 prompt。强调"字节稳定"是 cache 命中的工程化前提——时间戳、UUID、动态内容都得放 append-only 区。

路由改进 · Feature Demand Routing

当前路由："按难度路由"——难任务用强模型，简单任务用弱模型。

本文建议：按"feature demand"路由

强模型（≥ floor）

  • 子代理（sub-agent）
  • 多步规划
  • 工具链编排
  • 反射 / 自评

→ 把杠杆换成能力

弱模型（< floor）

  • 接地 Q&A（grounded QA）
  • 检索增强生成
  • 结构化抽取
  • 格式转换

→ 发挥 grounding 优势，避免编排负荷

路由维度从"难度"切到"能力需求 × 编排需求"——富 harness 留给强模型，弱模型做不需要 orchestration 的事。

这是一个对工程团队立刻可用的建议。把"难/易"换成"feature demand"——弱模型做 grounded QA 反而比做子代理更稳。

局限 · 红队

  • n = 6 模型太脆 · 一个 Qwen 3.6 净负（−0.031）点对 r=0.99 拉动不小；需要 12+ 模型 + bootstrap 置信带 + Spearman 报告。
  • Baseline 是裸 loop · 2024-style 裸循环 vs 2026 全套优化——无法区分"任意良好设计的 harness"与"Writer 这套特殊设计"。建议加 LangGraph / Claude Agent SDK / OpenAI Assistants 做三方对照。
  • Table 1 未跑测 · 六系统对比基于"公开文档 + 设计期源码研究"，论文诚实称之为"设计期比较而非 benchmark"。
  • Writer 利益相关 · 28 位作者全部 @writer.com；GA 决策恰好是 Palmyra X6（自家）+ Sonnet 4.6——机制清单可普适，跨模型表是公司 N=1。
  • Capability floor 因果未明 · Qwen 0.42 / Flash 0.45 是 harness 错配还是 judge 协议对小模型系统性偏低？需要"子代理 prompt 强化"消融。

立场观察：n=22 质量差不显著，论文主动说 p 不显著——这是诚实信号；统计上"决断性"来自方向一致而非 p < 0.05。

红队是这份卡片的核心价值之一。读者听完"决断性数字"必须立刻听到"但这 5 个洞"。立场观察放在最后：作者是诚实的工业从业者，不是营销。

Verdict · 推荐深入

Verdict

形式化（Eq. 2 + Eq. 4）+ 方法学（controlled swap）+ KPI（CPM）三件套 可立即借用； 跨模型数字结论在 n=6 与基线选择上 需要外部复现验证。

可立即借用

  • Eq. 2 五项分解 → 自己的 token 账单拆解
  • Eq. 4 有效输入价 → cache 工程目标
  • controlled swap 方法学 → A/B 平台
  • CPM KPI → 立刻加 dashboard 栏
  • Feature demand routing → 路由策略重排

需要外部复现

  • −41%/$、−44%/s、−38%/tokens 数字本身
  • r = 0.99 的 leverage 曲线（12+ 模型 bootstrap）
  • 三方 harness 对照（Writer vs 通用 vs 裸 loop）
  • Capability floor 的因果方向
  • Writer GA 决策的外部迁移性

推荐：在自己的 agent 平台把 Eq. 2 + Eq. 4 重画一遍，加一个第三方 harness 做第三 arm，用 CPM 替换"单任务成本"作为发布门槛。

结语：把"机制清单 + 公式 + KPI"与"跨模型表"分开看——前者可普适，后者是公司 N=1。推荐行动：把 Eq. 2/4 在自己平台重画一遍。
